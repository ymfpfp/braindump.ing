Rewriting blog in Haskell (notes on FP, parser combinators)

I took the liberty of rewriting my blog in Haskell! Here I'm going to leave some notes about writing the blog for myself.

First off, we have a Markdown parser; I find writing a Markdown parser to be a quick litmus test for reasoning about a new language. As such you may find this useful if you're learning Haskell.

Pregaming info:

Everything in Haskell is a function, specifically a curried function, that is, a function that takes exactly one parameter. For example:

max :: (Ord a) => a -> a -> a
max :: (Ord a) => a -> (a -> a)

max 4 5
(max 4) 5

(Here, Ord is a type class, representing that a is a type that has operations for ordering. It reminds me of interfaces in other languages.)

As you can see, max takes a value of a, returning a function that takes a value of a and returns a value of a.

Curried functions are useful because they let you partially apply functions, making a stronger case for first-class functions if you will. The root of this is lambda calculus; see Learn Lambda Calculus in Y Minutes.

There is extra syntax to facilitate order of operations, e.g.:

slug :: String -> String
slug =
  map (\c -> if c == ' ' then '-' else c)
    . filter (\c -> isAlphaNum c || c == ' ')
    . map toLower
    . trim

summary = trim $ intercalate " " (take 50 $ words $ Md.documentToText doc)

The first one is obvious, it resolves to nested functions, i.e. map . filter . map . trim resolves to map(filter(map(trim))), returning a composed function.

The second one is just the function application operator, i.e. trim $ intercalate " " (take 50 $ words $ Md.documentToText doc) resolves to trim (intercalate " " (take 50 (words (Md.documentToText doc)))) which can be annoying. In other words, you can use it to replace parentheses that would otherwise be there to notate order of operations.

The example above also makes use of lambdas, anonymous curried functions.

You may have heard of thunks. Haskell is a lazily-evaluated language (compared to something like Lisp). A thunk is just a deferred computation. For example, lists are just syntactical sugar for cons cells:

print $ head [1 + 2, 2 + 3, 3 + 4, 4 + 5]
(1 + 2) : (2 + 3) : (3 + 4) : (4 + 5) : []

head :: [a] -> a

head will return 1 + 2 as a thunk. print will force the thunk to be evaluated. Also of interest is the fact that every function is curried, so there's two functions waiting to be evaluated for each list item. Having lazy evaluation like this is usually good but of course it depends on the context.

Behind the scenes, thunks are most likely lazy closures.

See From Imperative To Functional: How to make the leap for more pregaming.

Parsing Markdown

Start out by defining:

type Document = [Block]

data Block
  = Paragraph [Inline]
  | Heading (Int, [Inline], Props)
  | Blockquote [Block]
  | OrderedList [[Block]]
  | UnorderedList [[Block]]
  | Image (String, String)
  | HorizontalBreak
  | CodeBlock (String, String)
  | Fragment [Inline]
  deriving Show

data Inline
  = Plain String
  | Italic [Inline]
  | Bold [Inline]
  | Code String
  | Link (String, [Inline])
  deriving Show

Using type is basically type aliasing. data actually introduces new types, made of multiple constructors. deriving lets us automatically derive instances of type classes for custom types. In this case we want to be able to show (log) these types.

parse :: String -> Document
-- Either return `[]` or the `[Block]` part of the final `Just (blocks, stream)`.
parse raw = maybe [] fst (runParser raw)

maybe is a built-in function:

data Maybe a
  = Nothing
  | Just a
maybe :: b -> (a -> b) -> Maybe a -> b

It takes a value of type b, a function that takes a value of type a and returns a value of type b, and a value that is wrapped in a Maybe. If this last value is wrapped in a Just constructor rather than Nothing, it gets passed to the function. In other words, a ? (f(a) :: b) : b.

fst is another function in the prelude; its type signature is fst :: (a, b) -> a. It takes a pair and returns the first item in that pair. This is useful because runParser has the following type signature:

-- Take a string, maybe return a value and the rest of the string.
runParser :: String -> Maybe (t, String)

Once we have this list of blocks, we want to convert them to HTML or text. This is pretty straightforward:

documentToHtml :: Document -> String
documentToHtml = foldr (\block html -> blockToHtml block ++ html) ""

documentToText :: Document -> String
documentToText = foldr (\block text -> blockToText block ++ text) ""

wrap :: String -> String -> String
wrap tag content = "<" ++ tag ++ ">" ++ content ++ "</" ++ tag ++ ">"

wrapWithProps :: String -> [(String, String)] -> String -> String
wrapWithProps tag props content =
  "<" ++ tag ++ concatMap format props ++ ">" ++ content ++ "</" ++ tag ++ ">"
  where
    format (k, v) = " " ++ k ++ "=\"" ++ v ++ "\""

blockToHtml :: Block -> String
blockToHtml block = case block of
  Paragraph inline -> wrap "p" $ concatMap inlineToHtml inline
  Heading (depth, inline, props) ->
    wrapWithProps ("h" ++ show depth) props (concatMap inlineToHtml inline)
  Blockquote blocks -> wrap "blockquote" $ concatMap blockToHtml blocks
  OrderedList items -> wrap "ol" $ concatMap (wrap "li" . documentToHtml) items
  UnorderedList items -> wrap "ul" $ concatMap (wrap "li" . documentToHtml) items
  Image (alt, src) -> wrapWithProps "img" [("alt", alt), ("src", src)] ""
  HorizontalBreak -> "<hr>"
  CodeBlock (language, code) ->
    wrap "pre" $ wrapWithProps "code" [("class", "language-" ++ language)] code
  Fragment inline -> concatMap inlineToHtml inline

blockToText :: Block -> String
blockToText block = case block of
  Paragraph inline -> concatMap inlineToText inline
  Heading (_, inline, _) -> concatMap inlineToText inline
  Blockquote blocks -> concatMap blockToText blocks
  OrderedList items -> concatMap documentToText items
  UnorderedList items -> concatMap documentToText items
  Image (alt, _) -> alt
  HorizontalBreak -> ""
  CodeBlock (_, code) -> code
  Fragment inline -> concatMap inlineToText inline

inlineToHtml :: Inline -> String
inlineToHtml inline = case inline of
  Plain s -> s
  Italic nested -> wrap "i" $ concatMap inlineToHtml nested
  Bold nested -> wrap "b" $ concatMap inlineToHtml nested
  Code code -> wrap "code" code
  Link (link, nested) ->
    wrapWithProps "a" [("href", link), ("target", "_blank")] $
      concatMap inlineToHtml nested

Folds

documentToHtml and documentToText both make use of a built-in function called foldr. Like the comment attests, folds let you reduce a data structure to a single value. There is foldl, which folds from the left:

-- `Num a` requires that type `a` is a instance of typeclass `Num`.
sum' :: (Num a) => [a] -> a
sum' xs = foldl (\acc x -> acc + x) 0 xs

We use foldr, which folds from the right. It does this by building a bunch of nested thunks, e.g. (f 0 (f 1 (f 2 z))) where z is the starting value. This can be useful in infinite lists where the binary function can return early and skip evaluating the second parameter, e.g. a function like &&.

foldl1 and foldr1 are functions that set the starting accumulator to the first item in the iterator.

Other higher-order built-in functions and useful tricks used here:

  • concatMap: Map a function and then join the results together.

  • We use case-based pattern matching here.

Parser combinators

As it turns out, one functional way of writing parsers is parser combinators. You combine parsers, which at the end of the day are just functions over a stream, to generate The Ultimate Parser, and then you run it. You could have invented parser combinators has a really good example:

function lit(c) {
  return function (input) {
    var r = inputRead(input);
    if (r == c) {
      inputAdvance(input, 1);
      return c;
    } else {
      return failure;
    }
  };
}

function or(parser0, parser1) {
  return function (input) {
    var result0 = parser0(input);
    if (result0 != failure) {
      return result0;
    }
    var result1 = parser1(input);
    if (result1 != failure) {
      return result1;
    }
    return failure;
  };
}

var parser = or(lit("a"), lit("b"));
var result = parser(input);

Haskell has some parser combinator libraries you can use, notably Parsec, by using a package manager like Cabal. For educational purposes I wrote my own.

A parser looks something like this:

newtype Parser t = Parser { runParser :: String -> Maybe (t, String) }

newtype is very similar to data, except that you're passing in exactly one constructor. This allows the type checker to treat the type as a unique type, but allows it to be erased by the compiler for performance reasons. In this case that constructor is runParser, a function that takes a string and returns the result wrapped in a Maybe in the case that there's any parsing error. This means that the function doesn't necessarily have to do any parsing, but of course semantically it seems like it should. Also note the generic type t, our parsers can return any type.

We can take advantage of some type classes that come in the prelude to construct some fundamental parser combinators:

Functors

Functors are types that serve as a context around contained values, allowing functions to be mapped over the value inside. For example, Maybe is one, where you can run a function on the value it's wrapping, e.g. fmap (* 2) (Just 200) results in Just 400.

This is exactly what we want, we want to be able to wrap a parser in another parser.

The Functor type class in Haskell asks you to define this function fmap:

class Functor f where
  fmap :: (a -> b) -> f a -> f b

instance Functor Parser where
  fmap apply parser = Parser (\s ->
    case (runParser parser) s of
      Nothing -> Nothing
      Just (inside, stream) -> Just (apply inside, stream))

Take a function apply, take another instance of the same functor (in this case, another parser) that returns type a, and return a functor that returns type b. The actual code returns a parser that runs the passed-in parser, and applies apply inside the context.

The infix operator for fmap is <$>. For example, parseParagraph, a parser for paragraphs, might look something like this:

parseParagraph :: Parser Block
parseParagraph = Paragraph <$> manyUntil (match "\n") parseInline

parseInline :: Parser Inline

parseInline returns Inline as the contained value, but Paragraph is a constructor of Block, so observe that parseParagraph returns Block as the contained value. We don't want to return Block directly, but we do want to convert that contained value to Block, so we use fmap.

Applicative functors

Applicative functors are functors that wrap around functions. This means you can map the function inside one functor to another functor, thus why it's called "applicative". It's like chaining two values together by applying the function in one to another. Implementing the type class looks like this:

instance Applicative Parser where
  -- A parser that takes a value and returns it, but wrapped in the `Parser` context.
  pure x = Parser (\stream -> Just (x, stream))

  -- Apply, e.g. `Just (+3) <*> Just 4` will apply the function inside `Just` on lhs
  -- to the inner value inside `Just` on rhs.
  --
  -- `*>` and `<*` will be implemented from this, and these are implemented
  -- automatically with `<$>` and `<*>` for primitive chaining that ignores the
  -- context outside, e.g.
  --
  -- `Just 3 <* Just 5` becomes
  -- `(\a _ -> a) <$> (Just 3) <*> (Just 5)`, as Haskell implements `<*` as
  -- `(\a _ -> a) <$> a <*> b`.
  --
  -- Thus we are able to execute both functions and return the result of the first
  -- which is pretty cool.
  --
  -- The difference between `*>` and `<*` and monadic chaining is that applicative
  -- chaining means you have no way to change the "shape" of the result.
  pf <*> px = Parser (\stream -> 
    case (runParser pf) stream of
      Nothing -> Nothing
      Just (f, fs) -> case (runParser px) fs of
        Nothing -> Nothing
        Just (x, xs) -> Just (f x, xs))

This type class asks us to implement pure, and then the applicative apply operator. The typeclass already has default implementations for the left and right sequencing operators, which allow us to evaluate both functions but only return the result of one or the other. You can imagine this is useful for chaining together parsers.

Later on the comment about monadic chaining will be clarified.

Along with implementing the Functor typeclass, this lets us write a parser like lit in the JS example:

type SatisfyFunction = Char -> Bool
satisfy :: SatisfyFunction -> Parser Char

(:) :: a -> [a] -> [a]

match :: String -> Parser String
match [] = pure []
-- We want to `fmap (:)` over the match so we can prepend it to `match`, which means
-- this is recursive.
--
-- As for `<*>`, we take the function in the context on the lhs, and apply it to the
-- value in the context on the rhs.
match (x:xs) = (:) <$> satisfy (\c -> c == x) <*> match xs

We use head-tail pattern matching to get x and xs, and then we fmap (:) to the contained value of satisfy (\c -> c == x) to get a function as the contained value; in other words, the first half (:) <$> (satisfy :: Parser Char) produces a Parser ([Char] -> [Char]). Then we can apply this contained function with the applicative operator to match xs1(1) If it isn't obvious, type String = [Char]., which performs this recursively and joins the matched characters together.

In this case, if we get an empty string we return an empty match. Otherwise, if the whole string is matched - that is, none of the parsers produce a Nothing, then we return the whole match.

Alternative

This is a type class in the prelude that represent computations that can fail. Notably, it provides <|>, which is basically or in the JS example above.

instance Alternative Parser where
  -- A parser that takes a value and returns `Nothing`. Useful for erroring out.
  empty = Parser (\_ -> Nothing)

  -- A parser combinator that performs || on the results of two parsers.
  px <|> py = Parser (\stream ->
    case (runParser px) stream of
      Nothing -> (runParser py) stream
      Just pz -> Just pz)

  -- A parser combinator that turns a parser that parses 1 thing into one that
  -- parses 0+, always returning a list.
  many :: Parser t -> Parser [t]
  many parser = Parser (\stream ->
    case (runParser parser) stream of
      Nothing -> Just ([], stream)
      Just (x, xs) ->
        case (runParser $ many parser) xs of
          -- This is an unreachable case.
          Nothing -> Nothing
          Just (y, ys) -> Just (x:y, ys))

  -- 1+.
  some parser = Parser (\stream ->
    case (runParser parser) stream of
      Nothing -> Nothing
      Just (x, xs) ->
        case (runParser $ many parser) xs of
          -- This is an unreachable case.
          Nothing -> Nothing
          Just (y, ys) -> Just (x:y, ys))

Monads

... [Monads] are just beefed-up applicative functors, much like applicative functors are beefed-up functors. - Learn You A Haskell For Great Good

A monad is a monoid in the category of endofunctors.

It's nice that I can finally understand what this means2(2) Hm. This feels too easy.:

  • Endofunctor: We know a functor is a type that wraps around a value to provide a context and can be mapped over while keeping that context. An endofunctor is that, but with the extra definition that it can map to itself, e.g. Maybe a -> Maybe b. Anyways all functors as defined in the Haskell prelude are endofunctors.

  • Monoid: A type that enforces three properties:

    • An identity context, a base value.

    • A binary function.

    • Associativity. If the binary function is chained, which pairs are evaluated first should not matter.

    • A list is a monoid, for example: identity context is [], binary function can be ++, and associativity of the binary is indeed a property.

So this sentence, "a monad is a monoid in the category of endofunctors", just means that a monad is a type that enforces all the constraints of a monoid that can map over itself. What is the binary function for a monad though? It's join :: f (f a) -> f a which just merges the context. And you can see how that's an endofunctor, since we can map a monad over itself and use the binary function to get a third value. And that flattening is associative.

What makes monads special is that rather than just merging the context, it also provides a bind operation, i.e. bind :: f a -> (a -> f b) -> f b. This also merges the context3(3) I've also seen this described as "threading state through various computations"., but it is capable of returning a different type. Haskell represents this operation with the >>= operator.

I think an example of this "merging context" concept might be useful. From Learn You A Haskell For Great Good:

applyMaybe :: Maybe a -> (a -> Maybe b) -> Maybe b
applyMaybe Nothing f = Nothing
applyMaybe (Just x) f = f x

(Just 5) `applyMaybe` (\i -> Just $ show i)
Just 5 >>= (\i -> Just $ show i)

(You can wrap function names in backticks to use as infix.)

Yes Maybe is a monad, >>= has already been implemented in the prelude. Here we use monadic chaining to convert the contained value to a string while keeping the context. You can see that applyMaybe, which has the same type signature as >>=, is functionally the same.

Thus, a monad is "a monoid in the category of endofunctors"; its special power is that its binary function allows it to merge context through contained values of different types.

The magical part is that Haskell then provides some syntactical sugar in the form of do, which makes code seem almost imperative. For a practical example, we can take advantage of the fact that Maybe is a monad to simplify the implementation of the applicative apply operator from earlier:

pf <*> px = Parser (\stream -> 
  case (runParser pf) stream of
    Nothing -> Nothing
    Just (f, fs) -> case (runParser px) fs of
      Nothing -> Nothing
      Just (x, xs) -> Just (f x, xs))

pf <*> px = Parser (\stream -> 
  runParser pf stream >>= \(f, fs) -> 
  runParser px fs >>= \(x, xs) -> 
  Just (f x, xs))

pf <*> px = Parser (\stream -> do
  (f, fs) <- (runParser pf) stream
  (x, xs) <- (runParser px) fs
  Just (f x, xs))

We no longer need to handle the Nothing case because the Maybe bind operation handles that for us.

Using the >>= operator directly feels a bit jarring to me and I find the do syntax to be more intuitive. At the end of the day they're the same though, just another concept for composition if you will.

Earlier in this post I said that "[t]he difference between [applicative chaining] and monadic chaining is that applicative chaining means you have no way to change the 'shape' of the result". Hopefully after this entire section that makes more sense now.

Implementing the Monad type class looks like:

instance Monad Parser where
  return = pure

  (>>=) :: Parser a -> (a -> Parser b) -> Parser b
  parser >>= f = Parser (\stream ->
    case (runParser parser) stream of
      Nothing -> Nothing
      Just (x, xs) -> (runParser $ f x) xs)

There are two other methods of this type class but they're not particularly relevant and are derived from these two.

With monadic chaining, we can now write parsers that semantically look more iterative. A pretty good chunk of the parsers I wrote in fact make liberal use of the do syntax.

parseBlock :: Parser Block
parseBlock = do
  -- Parse as many newlines first.
  _ <- many $ satisfy (\c -> c == '\n')
  parseOrderedList <|> parseUnorderedList <|> parseCodeBlock <|>
   parseHorizontalBreak <|> parseImage <|> parseBlockquote <|>
    parseHeading <|> parseParagraph

parseHeading :: Parser Block
parseHeading = do
  depth <- length <$> some (match "#")
  _ <- match " " 
  nested <- manyUntil (match "\n") parseInline

  return $ Heading (depth, nested, [])

parseBlockquote :: Parser Block
parseBlockquote = do
  -- We can start by parsing each line starting with "> " from this point onward,
  -- and then joining them together to use that as the nested stream.
  content <- unlines <$> some parseLines
  case (runParser $ many parseBlock) content of
    Nothing -> empty
    Just (blocks, _) -> return $ Blockquote blocks

  where
  parseLines :: Parser String
  parseLines = do
    _ <- many $ match " "
    _ <- match ">"
    content <-
      (match " " *> manyUntil (match "\n") (satisfy $ \c -> c /= '\n') <* match "\n")
      <|> (match "\n" *> pure "<br>")
    return content

With these building blocks we can basically write a parser, as you can see with parseBlock. You can see the rest of the parsers in Markdown.hs.

Now here's the actual parse function:

parse :: String -> Document
-- Either return `[]` or the `[Block]` part of the final `Just (blocks, stream)`.
parse raw = maybe [] fst ((runParser $ many parseBlock) raw)

Static site generation

Onto static site generation.

I extended the parser to allow for extensions:

type Extension = Document -> (Document, Map String Document)

parseWithExtensions :: String -> [Extension] -> (Document, Map String Document)
parseWithExtensions raw [] = (parse raw, Map.empty)
parseWithExtensions raw extensions =
  foldl (\(doc, extras) extension -> 
    let (out, extra) = extension doc in (out, Map.union extras extra))
    (parse raw, Map.empty) extensions

The let in syntax is similar to where, just scoped to an expression.

This is useful for table of contents, frontmatter, footnotes, etc. E.g. frontmatter:

import Data.List (dropWhileEnd)
import qualified Markdown as Md

trim :: String -> String
trim = dropWhileEnd isSpace . dropWhile isSpace

parseFrontmatter :: Md.Extension
parseFrontmatter doc = case doc of
  Md.HorizontalBreak : rest
    | (fields, Md.HorizontalBreak : body) <- break isBreak rest ->
      (body, Map.fromList (concatMap toEntry fields))
  _ -> (doc, Map.empty)

  where
    isBreak :: Md.Block -> Bool
    isBreak Md.HorizontalBreak = True
    isBreak _ = False

    toEntry :: Md.Block -> [(String, Md.Document)]
    toEntry (Md.Paragraph inline) =
      case break (== ':') (concatMap Md.inlineToText inline) of
        (key, ':' : value) -> [(trim key, [Md.Fragment [Md.Plain $ trim value]])]
        _ -> []
    toEntry _ = []

I wanted to point this out because this uses pattern matching in the form of <- and ->. break takes a a -> Bool and will break the provided list in half once it reaches an item that validates the function. In this case, we split at the next horizontal break and then return the rest of the body and the parsed fields. If this doesn't work it just falls through as one would expect.

I/O monad

To actually interact with the real world and do Real World things, like perform static site generation, Haskell has the IO monad. For example, to read from a file and pass it to the parser:

main :: IO ()
main = do
  raw <- readFile "../routes/index.md"
  let
    (parsed, _) = Markdown.parseWithExtensions raw []
  putStrLn $ Markdown.documentToHtml parsed

Since monads merge the context, this lets you thread I/O actions together.

I was curious what happens behind the scenes with I/O chaining, since these monads are essentially what cross over the boundary between pure functions and functions with side effects. Unraveling the mystery of the IO monad is a good resource for this4(4) TODO to self, will come back and fill this out. .

The code with side effects (in Main.hs) is somewhat messy and I have yet to clean it up but it fundamentally does this:

import Data.List (isSuffixOf)
import System.Directory (
  copyFile,
  createDirectoryIfMissing,
  doesDirectoryExist,
  listDirectory)

import qualified Markdown as Md
import qualified Extensions as Extensions

main :: IO ()
main = do
  createDirectoryIfMissing True outputDirectory

  -- Output all assets.
  outputDirectoryTo assetsDirectory outputDirectory [".html", ".ts"]

  -- Now begin to process the actual Markdown content.
  outputDirectoryTo inputDirectory outputDirectory []

  where
  outputDirectoryTo :: String -> String -> [String] -> IO ()
  outputDirectoryTo input output blacklist = do
    entries <- listDirectory input
    -- mapM_ :: Monad m => (a -> m ()) -> [a] -> m ()
    -- mapM_ :: Monad IO => (String -> IO ()) -> [String] -> IO ()
    mapM_ (\entry -> do
      let path = input ++ entry
      let outputPath = output ++ entry
      isDir <- doesDirectoryExist path
      case isDir of
        True -> do
          createDirectoryIfMissing True outputPath
          outputDirectoryTo (path ++ "/") (outputPath ++ "/") blacklist
        False -> when (not $ any (\suffix -> suffix `isSuffixOf` entry) blacklist) $ do
          -- Check if there's any transformations that need to happen.
          case Map.lookup (extension entry) transformations of
            Nothing -> copyFile path outputPath
            Just f -> do
              (transformedPath, transformed) <- f path
              writeFile (output ++ transformedPath) transformed) entries

Note that this uses mapM_, which lets us map items to a monadic action while ignoring the results. mapM is similar but actually collects the results.
